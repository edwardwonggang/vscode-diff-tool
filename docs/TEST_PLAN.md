# Remote Git Diff 测试方案

## 目标

验证插件在真实远程 Linux Git 仓库上的可用性、稳定性和性能，重点覆盖以下用户体验风险：

- 能否稳定识别远程工作区变更
- 点击任意变更文件时能否稳定打开对比
- 快速切换文件时旧请求是否会拖慢新请求
- 大文件和大改动文件是否会走合适的预览策略
- 断连、重连、空闲回收后是否还能继续使用
- 测试结束后是否能清理后台进程，避免污染下一轮测试

## 场景矩阵

| 场景 | 风险 | 当前覆盖方式 |
| --- | --- | --- |
| 首次启动自动连接 | 配置读取失败、仓库校验失败 | 自动化 |
| 刷新变更列表 | 远程 `git status` 慢或失败 | 自动化 |
| 目录树展示 | 同目录/跨目录切换时节点状态异常 | 自动化 + 人工观察 |
| 普通文本文件左右对比 | 原生 diff 可用但速度慢 | 自动化 |
| 大文件少量变更 | 原生 diff 可接受但不能错误切到降级视图 | 自动化 |
| 中等文件超大量变更 | 误走原生 diff 导致长时间卡住 | 自动化 |
| 超大文件超大量变更 | 需要补丁视图兜底 | 自动化 |
| 二进制文件点击 | 不应进入文本 diff | 自动化 |
| 快速连续切换文件 | 旧请求拖住新请求 | 自动化 |
| 空闲后再点击文件 | SSH 空闲回收后恢复失败 | 自动化 + 后续补充 |
| 手动断开再重连 | 状态机错乱、缓存未清理 | 自动化 |
| 认证错误/路径错误 | 错误提示是否准确且仅在错误时提示 | 文档化，后续可扩展自动化 |

## 自动化用例

### A. 启动级全量自测

入口：`npm run visual-test`

插件在独立 VS Code 实例中自动执行：

1. 打开插件视图
2. 自动刷新变更列表
3. 遍历当前项目全部 tracked 变更文件
4. 对每个文件记录：
   - 文件路径
   - 文本/二进制类型
   - 新增/删除行数
   - 打开耗时
5. 对文本文件执行打开预览
6. 对二进制文件验证“不可对比”分支
7. 对前若干文本文件执行快速切换压力测试
8. 输出项目级汇总和明细报告
9. 终止本次测试启动的 VS Code 进程树

### B. 纯逻辑单元测试

入口：`npm test`

当前覆盖：

- `shouldUsePatchView`
  - 普通大文件但变更不大，不应切补丁视图
  - 中等大小但变更行数极大，应切补丁视图
  - 真实慢文件样本 `.c` / `.h` 阈值命中
- `buildPatchRows`
  - patch 左右行对齐逻辑
- `InteractionController`
  - 最新请求判定
  - session 失效逻辑

## 性能关注点

重点记录以下阶段耗时：

- `statsMs`
- `sizeMs`
- `patchMs`
- `headReadMs`
- `worktreeReadMs`
- `nativeDiffMs`
- `totalOpenMs`

目标不是单纯追求所有文件都走原生 diff，而是：

- 普通文本文件优先原生左右对比
- 大变更文件尽早切换到补丁视图
- 用户快速切换时，旧请求不能继续大量占用资源拖慢当前请求

## 通过标准

- 全量自测完成且无 `启动自测失败`
- 所有 tracked 变更文件都被统计到报告
- 二进制文件不会尝试打开文本 diff
- 快速切换压力测试能够完成
- 本次测试结束后无残留测试 VS Code 进程
- `test-results/visual-report.md` 和 `test-results/visual-report.json` 生成成功

## 当前已识别的高风险点

1. 原实现会在决定视图模式之前直接读取 HEAD 和工作区全文。
2. 当用户快速切换文件时，旧请求虽然在 UI 层失效，但底层全文读取仍继续执行。
3. “中等文件 + 超大量变更”此前未命中补丁视图阈值，会被错误交给原生 diff，导致几十秒卡顿。
4. 大文件打开慢不一定是文件体积问题，更多是变更行数和读取策略共同造成。
